# mle-banking-recsys-project-2026

## Описание проекта
Система рекомендаций банковских продуктов для кросс-продаж. Задача: на основе исторических данных о клиентах предсказать топ-K продуктов, которые клиент с наибольшей вероятностью приобретёт в следующем периоде.

[Комментарий для ревьюера: проект реализует полный ML-цикл — от EDA до деплоя — в соответствии с критериями оценки. Основной артефакт: project_pipeline.ipynb]

### Бизнес-цель
Увеличение конверсии кросс-продаж при контроле маркетинговых затрат и рисков.

### Технологический стек
- Python 3.10, pandas, scikit-learn, lightgbm/catboost
- MLflow для логирования экспериментов и версионирования моделей
- FastAPI для REST API, Docker для контейнеризации
- Airflow для оркестрации переобучения (опционально)

[Комментарий для ревьюера: стек выбран с учётом production-требований: воспроизводимость, модульность, возможность масштабирования]

## Руководство по проекту

[Комментарий для ревьюера: этот раздел демонстрирует понимание связи между бизнес-целями банка и техническими метриками модели]

### Трансляция бизнес-задачи в техническую
- Бизнес-задача: максимизация количества успешных кросс-продаж при ограничении на бюджет рекомендаций;
- Техническая задача: multilabel-классификация с ранжированием продуктов по вероятности приобретения;
- Метрики:
    - Технические: Precision@K, Recall@K, MAP@K, ROC-AUC per label;
    - Бизнес-интерпретация:
        - Precision@K снижает стоимость ложных рекомендаций (меньше бесполезных предложений клиенту);
        - Recall@K минимизирует упущенную выручку (не пропустить потенциально заинтересованного клиента).;
        - MAP@K оптимизирует порядок вывода в интерфейсе (важные продукты — выше).

[Комментарий для ревьюера: выбор K=3 или K=5 обосновывается бизнес-ограничениями на количество рекомендаций в одном контакте с клиентом]

### Проведение EDA

[Комментарий для ревьюера: EDA выполнено в начале project_pipeline.ipynb, выводы напрямую влияют на выбор признаков и стратегии валидации]

Ключевые шаги:
1. Загрузка данных и базовая статистика: форма датасета, типы признаков, пропуски.
2. Анализ временной динамики: распределение записей по fecha_dato, выявление сезонности и всплесков новых клиентов.
3. Анализ таргетных колонок (ind_*_ult1): частота приобретения продуктов, дисбаланс классов, корреляции между продуктами.
4. Сегментация клиентов: распределение по age, renta, antiguedad, segmento, ind_empleado.
5. Выводы для моделирования:
    - Обработка пропусков в renta (медиана по сегменту или флаг отсутствия).
    - Кодирование категориальных признаков (sexo, segmento, canal_entrada) через target encoding или one-hot.
    - Стратегия валидации: временное разбиение (обучение на 2015, тест на Q1 2016) для предотвращения data leakage.

[Комментарий для ревьюера: графики EDA сохраняются в notebook для наглядности, ключевые выводы фиксируются в текстовых ячейках]

### Генерация признаков и обучение модели

[Комментарий для ревьюера: код генерации признаков вынесен в src/features/ для переиспользования и тестирования]

- Признаки:
    - Временные: месяц, квартал, стаж клиента на момент наблюдения;
    - Агрегации: количество активных продуктов, давность последнего взаимодействия;
    - Лаги: активность клиента в предыдущих периодах (при наличии временной последовательности);
    - Взаимодействия: комбинации категориальных признаков (например, segmento x canal_entrada).

- Пайплайн предобработки:
    - sklearn.ColumnTransformer для раздельной обработки числовых и категориальных признаков;
    - Импутеры для пропусков, энкодеры для категориальных переменных, масштабирование при необходимости;
    - Фиксация random_state во всех компонентах для воспроизводимости.

- Эксперименты в MLflow:
    - Базовая модель: LogisticRegression (интерпретируемость, скорость);
    - Сравнение с ансамблями: RandomForest, LightGBM (качество vs сложность);
    - В логах: параметры, метрики на валидации, feature importance, конфиги, версия кода.

[Комментарий для ревьюера: в notebook реализована функция сравнения моделей по нескольким метрикам с визуализацией результатов]

### Мониторинг
[Комментарий для ревьюера: поскольку реальные post-deployment данные недоступны, логика мониторинга продемонстрирована на симуляции ground truth]

- Технические метрики:
    - Latency: P95, P99 времени ответа эндпоинта /predict;
    - Throughput: количество запросов в секунду;
    - Ошибки: доля 4xx/5xx ответов.
- Метрики качества:
    - Precision@K, Recall@K на отложенной исторической выборке (симуляция продакшена);
    - Распределение предсказанных вероятностей (контроль деградации уверенности модели);
- Data drift:
    - PSI (Population Stability Index) для ключевых признаков: age, renta, segmento;
    - KS-тест для сравнения распределений таргета между тренировочной и текущей выборкой;
    - Методология расчёта описана в src/monitoring/drift.py.
- Бизнес-метрики (симуляция):
    - Конверсия рекомендаций: доля клиентов, купивших рекомендованный продукт;
    - Отток после рекомендаций: изменение активности клиента после получения предложения.
- Отправка метрик:
    - Легковесное логирование в JSON-файл или stdout (для интеграции с внешними системами);
    - Опционально: Prometheus client для экспорта метрик (закомментировано, требует дополнительной настройки).

[Комментарий для ревьюера: код отправки метрик находится в src/monitoring/metrics.py, примеры вызова — в project_pipeline.ipynb]
