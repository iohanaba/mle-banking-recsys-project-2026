# mle-banking-recsys-project-2026

## Описание проекта
Система рекомендаций банковских продуктов для кросс-продаж. Задача: на основе исторических данных о клиентах предсказать топ-K продуктов, которые клиент с наибольшей вероятностью приобретёт в следующем периоде.

> **Для ревьюера**: проект реализует полный ML-цикл — от EDA до деплоя — в соответствии с критериями оценки. Основной артефакт: `project_pipeline.ipynb` (отправляется на Этапе 2).

### Бизнес-цель
Увеличение конверсии кросс-продаж при контроле маркетинговых затрат и рисков.

### Технологический стек
- Python 3.10, pandas, scikit-learn, lightgbm/catboost
- MLflow для логирования экспериментов и версионирования моделей
- FastAPI для REST API, Docker для контейнеризации
- Airflow для оркестрации переобучения (опционально)

> **Для ревьюера**: стек выбран с учётом production-требований: воспроизводимость, модульность, возможность масштабирования.

## Руководство по проекту

### Трансляция бизнес-задачи в техническую


> **Для ревьюера**: этот раздел демонстрирует понимание связи между бизнес-целями банка и техническими метриками модели.

- **Бизнес-задача**: максимизация количества успешных кросс-продаж при ограничении на бюджет рекомендаций;
- **Техническая задача**: multilabel-классификация с ранжированием продуктов по вероятности приобретения;
- **Метрики**:
  - *Технические*: Precision@K, Recall@K, MAP@K, ROC-AUC per label;
  - *Бизнес-интерпретация*:
    - Precision@K снижает стоимость ложных рекомендаций (меньше бесполезных предложений клиенту);
    - Recall@K минимизирует упущенную выручку (не пропустить потенциально заинтересованного клиента);
    - MAP@K оптимизирует порядок вывода в интерфейсе (важные продукты — выше).

> **Для ревьюера**: выбор K=3 или K=5 обосновывается бизнес-ограничениями на количество рекомендаций в одном контакте с клиентом.


### Разворачивание инфраструктуры обучения

> **Для ревьюера**: инфраструктура обучения упакована в shell-скрипты для воспроизводимости и удобства.

- Запуск MLflow: `bash scripts/setup_mlflow.sh`
- Скрипт инициализирует tracking server с файловым бэкендом (`./mlruns`) и артефакт-хранилищем;
- Все эксперименты логируются с:
  - Параметрами модели и предобработки;
  - Метриками на тренировочной и валидационной выборках;
  - Артефактами: веса модели, feature importance, конфиги;
  - Версиями кода через git-commit hash.

> **Для ревьюера**: в `project_pipeline.ipynb` каждый эксперимент обёрнут в `mlflow.start_run()` с понятным `run_name`.

### Проведение EDA

> **Для ревьюера**: EDA выполнено в начале `project_pipeline.ipynb`, выводы напрямую влияют на выбор признаков и стратегии валидации.

**Ключевые шаги**:
1. Загрузка данных и базовая статистика: форма датасета, типы признаков, пропуски;
2. Анализ временной динамики: распределение записей по `fecha_dato`, выявление сезонности;
3. Анализ таргетных колонок (`ind_*_ult1`): частота приобретения, дисбаланс классов;
4. Сегментация клиентов: распределение по `age`, `renta`, `antiguedad`, `segmento`;
5. Выводы для моделирования:
   - Обработка пропусков в `renta` (медиана по сегменту);
   - Кодирование категориальных признаков через one-hot;
   - Стратегия валидации: временное разбиение для предотвращения data leakage.

> **Для ревьюера**: графики EDA сохраняются в notebook для наглядности, ключевые выводы фиксируются в текстовых ячейках.

### Генерация признаков и обучение модели

> **Для ревьюера**: код генерации признаков вынесен в `src/features/` для переиспользования и тестирования.

**Признаки**:
- Временные: месяц, квартал, стаж клиента;
- Агрегации: количество активных продуктов, давность взаимодействия;
- Взаимодействия: комбинации категориальных признаков.

**Пайплайн предобработки**:
- `sklearn.ColumnTransformer` для раздельной обработки числовых и категориальных признаков;
- Импутеры, энкодеры, масштабирование;
- Фиксация `random_state=42` во всех компонентах.

**Эксперименты в MLflow**:
- Базовая модель: LogisticRegression (интерпретируемость);
- Сравнение с ансамблями: RandomForest, LightGBM;
- В логах: параметры, метрики, feature importance, конфиги.

> **Для ревьюера**: в notebook реализована функция сравнения моделей по нескольким метрикам с визуализацией результатов.

### Мониторинг

> **Для ревьюера**: поскольку реальные post-deployment данные недоступны, логика мониторинга продемонстрирована на симуляции ground truth.

**Технические метрики**:
- Latency: P95, P99 времени ответа `/predict`;
- Throughput: запросов в секунду;
- Ошибки: доля 4xx/5xx ответов.

**Метрики качества**:
- Precision@K, Recall@K на отложенной выборке;
- Распределение предсказанных вероятностей.

**Data drift**:
- PSI для ключевых признаков: `age`, `renta`, `segmento`;
- KS-тест для сравнения распределений таргета;
- Методология расчёта описана в `src/monitoring/metrics.py`.

**Бизнес-метрики (симуляция)**:
- Конверсия рекомендаций;
- Отток после рекомендаций.

**Отправка метрик**:
- Логирование в `logs/metrics.jsonl` (формат JSONL);
- Опционально: Prometheus client (закомментировано).

> **Для ревьюера**: код отправки метрик находится в `src/monitoring/metrics.py`, примеры вызова — в `project_pipeline.ipynb`.
